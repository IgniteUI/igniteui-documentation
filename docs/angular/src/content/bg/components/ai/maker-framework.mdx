# MAKER Framework

MAKER Framework (`@igniteui/maker-mcp`) е многоагентен сървър за оркестрация на AI MCP от Infragistics, който разлага сложни задачи на валидирани, изпълними планове на стъпки, използвайки алгоритъм на базата на консенсус гласуване между множество AI агенти. MAKER означава Maximal Agentic decomposition, first-to-ahead-by-K Error correction и Red-flagging. Framework-ът се основава на научната статия *Solving a million-step LLM task with zero errors* от Cognizant AI Lab. Той работи като MCP сървър чрез `npx` от регистъра `@igniteui` GitHub Packages и се свързва към всеки MCP-съвместим AI клиент чрез STDIO транспорт. След свързване AI асистентът може да извика три инструмента - `plan`, `execute` и `plan_and_execute` - за да изпълни задачи с дълъг хоризонт с автоматично открояване и коригиране на грешки.

MAKER Framework не е инструмент за скеле на компоненти на Ignite UI. За създаване на проекти Ignite UI, генериране на компоненти и запитвания към документация, използвайте [CLI MCP сървъра](cli-mcp.md). MAKER е независим от framework-а - той не насочва специално Angular, React, Blazor или Web Components и сам не чете или модифицира файлове на изходния код на проекта. Изисква поне един ключ на AI доставчик (OpenAI, Anthropic или Google AI) и GitHub Personal Access Token със `read:packages` обхват за регистъра `@igniteui`.

## Как работи MAKER

MAKER Framework работи в две последователни фази: планиране и изпълнение. Всяка фаза използва набор от вътрешни AI клиенти и прилага един и същ механизъм на консенсус гласуване за валидиране на резултатите преди да преминат напред.

В **фазата на планиране**, клиент за планиране предлага партиди от стъпки (конфигурируем размер на партидата, по подразбиране 3). След това отделен клиент за гласуване на плана изпълнява гласуване "first-to-ahead-by-K": множество AI агенти гласуват "Да", "Не" или "Край" за предложените стъпки. Предложение се приема, когато един резултат води с K гласа пред другите. Отхвърлени предложения включват причините за отхвърляне в следващия повтор, което движи непрекъснато усъвършенстване, докато планът е завършен или е достигнат лимитът на повторите.

В **фазата на изпълнение**, framework-ът обработва валидирания план на стъпки в конфигурируеми партиди. Клиент за изпълнение изпълнява всяка партида и произвежда актуализирано състояние. Клиент за гласуване на изпълнение валидира новото състояние спрямо предишното състояние и оригиналната задача, използвайки един и същ K-марж алгоритъм за гласуване. Отхвърлени изпълнения се повтарят с обратна информация. Крайното натрупано състояние е резултатът на задачата.

Слоят на валидиране на червения флаг работи през двете фази. Всяко искане на AI клиент е обвито в защитен разговор, който изпълнява конфигурируеми валидатори на отговора, преди той да влезе в системата за гласуване. Деформирани или твърде кратки отговори се повтарят със съобщението за неуспех, добавено към подсказката. Персонализирани валидатори могат да бъдат добавени чрез имплементиране на `IAIRedFlagValidator`.

## Инсталиране на MAKER MCP сървъра

MAKER е публикуван на GitHub Packages под обхвата `@igniteui` и изисква еднократна настройка на регистъра. Изпълнете следната команда веднъж на вашата машина:

```bash
npm config set @igniteui:registry https://npm.pkg.github.com
```

След това отворете `~/.npmrc` (Windows: `%USERPROFILE%\.npmrc`) и добавете вашия GitHub Personal Access Token със `read:packages` обхват:

След конфигурирането на регистъра, MCP сървърът работи чрез `npx` без отделна глобална инсталация. Нативният двоичен файл (\~50 MB) се изтегля и кешира при първи старт. Следващите стартове са мигновени.

## Свързване към вашия AI клиент

Добавете записа на сървъра `maker` към конфигурационния файл на MCP за вашия AI клиент. Блокът `env` пропуска вашия ключ на доставчик на AI; задайте ключове само за доставчиците, които използвате.

### VS Code (GitHub Copilot)

Добавете към `.vscode/mcp.json` в вашето работно пространство:

```json
{
  "servers": {
    "maker": {
      "command": "npx",
      "args": ["-y", "@igniteui/maker-mcp", "--stdio"],
      "env": {
        "Executor__AIProviderKeys__OpenAI": "<your-openai-key>"
      }
    }
  }
}
```

Преместете Copilot Chat в режим Агент и потвърдете, че `plan`, `execute` и `plan_and_execute` се появяват в списъка на инструментите.

### Claude Desktop

1. Отворете вашия конфигурационен файл на Claude Desktop:
   - macOS: `~/Library/Application Support/Claude/claude_desktop_config.json`
   - Windows: `%APPDATA%\Claude\claude_desktop_config.json`

2. Добавете записа `maker` към блока `mcpServers`:

   ```json
   {
  "mcpServers": {
    "maker": {
      "command": "npx",
      "args": ["-y", "@igniteui/maker-mcp", "--stdio"],
      "env": {
        "Executor__AIProviderKeys__Anthropic": "<your-anthropic-key>"
      }
    }
  }
}
   ```

3. Рестартирайте Claude Desktop. Първият старт изтегля нативния двоичен файл (\~30 с при типична връзка).

### Cursor, Claude Code, JetBrains и други MCP клиенти

Всеки клиент, който поддържа MCP с STDIO транспорт, използва един и същ блок `mcpServers`:

```json
{
  "mcpServers": {
    "maker": {
      "command": "npx",
      "args": ["-y", "@igniteui/maker-mcp", "--stdio"],
      "env": {
        "Executor__AIProviderKeys__OpenAI": "<your-openai-key>"
      }
    }
  }
}
```

## Конфигуриране на MAKER MCP сървъра

Цялата конфигурация на MAKER се подава чрез променливи на среди. Конвенцията за наименование използва двойното подчертаване (`__`) като разделител на раздели. Преминете променливи в блока `env` на конфигурацията на вашия MCP клиент.

### Ключове на доставчици на AI

Задайте поне един ключ на доставчик. Можете да задавате ключове за множество доставчици одновременно и да приказвате различни доставчици на различни вътрешни клиенти.

| Променлива на среди | Описание       |
| -------------------- | ----------------- |
| `Executor__AIProviderKeys__OpenAI`       | Ключ на OpenAI API    |
| `Executor__AIProviderKeys__Anthropic`       | Ключ на Anthropic API |
| `Executor__AIProviderKeys__Google`       | Ключ на Google AI API |

### Изберете модели на вътрешен клиент

MAKER използва четири вътрешни AI клиенти: `Planning`, `PlanVoting`, `Execution` и `ExecutionVoting`. Всеки клиент може да насочи различен доставчик и модел. Подразбиранията използват `gpt-5.1` на OpenAI за всички четири клиенти.

За смесване на доставчици за баланс на разходи и качество, задайте `Provider` и `Model` променливи на клиент:

```json
"env": {
  "Executor__AIProviderKeys__OpenAI":             "<openai-key>",
  "Executor__AIProviderKeys__Anthropic":          "<anthropic-key>",
  "Executor__Clients__Planning__Provider":        "Anthropic",
  "Executor__Clients__Planning__Model":           "claude-opus-4-5",
  "Executor__Clients__PlanVoting__Provider":      "OpenAI",
  "Executor__Clients__PlanVoting__Model":         "gpt-4.1-mini",
  "Executor__Clients__Execution__Provider":       "Anthropic",
  "Executor__Clients__Execution__Model":          "claude-opus-4-5",
  "Executor__Clients__ExecutionVoting__Provider": "OpenAI",
  "Executor__Clients__ExecutionVoting__Model":    "gpt-4.1-mini"
}
```

Валидни `Provider` стойности: `OpenAI`, `Anthropic`, `Google`.

### Свържете външни MCP сървъри

MAKER може да се свърже към външни MCP сървъри като доставчици на инструменти, което прави техните инструменти достъпни за клиента за изпълнение. Конфигурирайте всеки сървър с име, описание, URL и опционален ключ на API, използвайки индексирани променливи на среди:

```json
"env": {
  "Executor__McpServers__0__Name":        "igniteui-cli",
  "Executor__McpServers__0__Description": "Ignite UI CLI scaffolding and documentation tools",
  "Executor__McpServers__0__Url":         "https://mcp.example.com/igniteui-cli"
}
```

Увеличете индекса (`0`, `1`, `2`, ...) за регистриране на допълнителни сървъри.

### Настройка на размера на партидата и праг на гласуване

Всеки разговор на инструмент на MCP приема `batchSize` и `k` параметри на естествен език. Задайте ги в вашата подсказка при извикване на инструментите.

| Параметър      | Подразбиране        | Описание                                                                                                            |
| -------------- | -------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `batchSize` | `3` | Стъпки, предложени или изпълнени по кръг на гласуване. По-ниски стойности дават повече контрол; по-висока стойност подобрява пропускателната способност.         |
| `k` | `10` | Праг на консенсус марж. По-висока стойност изисква по-силно съгласие между агентите за гласуване, преди резултат да бъде приет. |

По-ниска `k` (3-5) за по-бързи, изследователски задачи. Повишете `k` (15-20) за критични задачи, където коректност е по-важна от разходите на токени.

## Налични инструменти

MAKER MCP сървърът излага три инструмента на свързания AI агент.

| Инструмент           | Описание                                                                     |
| -------------- | ------------------------------------------------------------------------------- |
| `plan` | Разложете задача на валиден, подреден списък от стъпки без да го изпълнявате       |
| `execute` | Изпълнете списък от стъпки, произведен от `plan`, с консенсус гласуване на партида |
| `plan_and_execute` | Планирайте и изпълнете в един разговор с живи събития на напредък между фазите           |

Използвайте `plan` последван от `execute`, когато искате да проверите и валидирате плана на стъпките преди изпълнение. Използвайте `plan_and_execute` за безконтролни тичания, където проверката не е необходима.

## Обичайни работни потоци

### Еднократна задача

Извикайте `plan_and_execute` директно със подразбиранията:

> Използвайте plan\_and\_execute за писане на подробно сравнение на REST срещу GraphQL за технически блог пост.

### Инспектирайте плана преди изпълнението

Използвайте `plan` за преглед на стъпките, след това ги предайте на `execute`:

> 1. Използвайте plan за създаване на план за миграция на PostgreSQL схема на многоклиентски дизайн.
> 2. Покажи ми стъпките.
> 3. Използвайте execute с тези стъпки за изпълнение на скрипта на миграцията.

### Изпълнение с висока увереност за критични задачи

Повишете `k` за да изискате по-силен консенсус:

> Използвайте plan\_and\_execute с batchSize=3 и k=15 за преглед на този смарт контракт за уязвимости в сигурност.

### Евтина изследователска задача

Намалете и двата параметъра, за да минимизирате използването на токени:

> Използвайте plan\_and\_execute с batchSize=3 и k=3 за проект на README на проект за Node.js CLI инструмент.

## Поддържани платформи

Нативният двоичен файл се дистрибутира за четири платформи. Правилният двоичен файл се избира и кешира автоматично при първи старт.

| Платформа | Архитектура          | RID            |
| -------- | --------------------- | -------------- |
| Windows  | x64                   | `win-x64` |
| macOS    | x64 (Intel)           | `osx-x64` |
| macOS    | arm64 (Apple Silicon) | `osx-arm64` |
| Linux    | x64                   | `linux-x64` |

ARM Linux не е в момента пакетиран. За да поискате подкрепа на допълнителни платформи, отворете проблем на [github.com/IgniteUI/MAKER](https://github.com/IgniteUI/MAKER).

Местоположението на кеша на двоичния файл може да бъде заменено с променливата на среди `MAKER_MCP_CACHE`. Пътищата на кеша по подразбиране са `%LOCALAPPDATA%\maker-mcp\{version}\{rid}\` на Windows и `~/.cache/maker-mcp/{version}/{rid}/` на macOS и Linux.

## Bootstrap グリッド定義

{ProductName} には、Bootstrap グリッドのような強力な flex ベースのレイアウト システムが含まれています。現在のあらゆるモダン アプリケーションはレスポンシブ Web デザイン アプローチに従うことが期待されます。つまり、デバイス サイズに基づいて、または単にブラウザーのサイズ変更によって、HTML 要素のレイアウトを適切に調整できます。以前は Bootstrap グリッド レイアウトが最も使用されていたアプローチでしたが、CSS グリッドのような flex ベースのレイアウト システムは任意のブラウザーで動作するため、より一般的になっています。{ProductName} Layout Directive は、コンテンツ / テキストの折り返し、均等配置、配置を含む、垂直方向および水平方向のフローを可能にします。{ProductName} グリッドは CSS を使用したレスポンシブ レイアウトをサポートし、サイズ変更時のグリッドの動作に究極の柔軟性を提供します。

## 編集可能な {Platform} グリッド

グリッド編集の各操作にはバッチ操作が含まれます。つまり、API により、編集を単一のサーバー呼び出しにグループ化するオプションが提供されるか、グリッド操作に伴ってグリッドの編集 / 更新操作を実行できます。CRUD 操作を備えた編集可能なグリッドとして優れた開発者エクスペリエンスを提供するだけでなく、グリッドには Excel のようなキーボード ナビゲーションが含まれています。一般的な既定のグリッド ナビゲーションが含まれており、さらに顧客のニーズに合わせて任意のナビゲーション オプションをオーバーライドするオプションもあります。優れたナビゲーション スキームを備えた編集可能なグリッドは、現代のあらゆる業務アプリケーションに不可欠です。Ignite UI グリッドを使用すれば、これを簡単に実現できます。

このトピックに従って、[セル テンプレート](data-grid.md#cell-template)、[セル編集テンプレート](data-grid.md#cell-editing-template)、および編集について詳しく学習できます。

```typescript
constructor() {
    var grid = this.grid = document.getElementById('grid') as IgcGridComponent;
    var name = this.name = document.getElementById('name') as IgcColumnComponent;
    var subscription = this.subscription = document.getElementById('subscription') as IgcColumnComponent;
    grid.data = this.data;
    name.bodyTemplate = this.nameCellTemplate;
    subscription.bodyTemplate = this.subscriptionCellTemplate;
}

public nameCellTemplate = (ctx: IgcCellTemplateContext) => {
    return html`
        <span tabindex="0" @keydown="${() => this.deleteRow(ctx.cell.id.rowIndex)}">${this.formatTitleCase(ctx.cell.value)}</span>
    `;
}

public subscriptionCellTemplate = (ctx: IgcCellTemplateContext) => {
    if (ctx.cell.value) {
            return html` <input type="checkbox" checked /> `;
    } else {
            return html` <input type="checkbox"/> `;
    }
}

public deleteRow(rowIndex: number) {
     this.grid.deleteRow(rowIndex);
}

public formatTitleCase(value: string) {
    return value.toLowerCase().replace(/\b\w/g, c => c.toUpperCase());
}
```

```javascript
function formatTitleCase(value) {
    return value.toLowerCase().replace(/\b\w/g, c => c.toUpperCase());
```

データ取得に関連するすべてのロジックを別のデータ サービスに分離することが推奨されるため、サーバーからのデータ取得を処理するサービスを作成します。

サービスを別ファイルに実装しましょう

```typescript
// northwind.service.ts

import { Injectable } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { Observable } from 'rxjs/Observable';
import { of } from 'rxjs/observable/of';
import { catchError, map } from 'rxjs/operators';
```

すべての {Platform} サービス定義における[必須要素](https://angular.io/guide/dependency-injection)である `Injectable` デコレーターをインポートしています。`HttpClient` はバックエンド サービスと通信するための機能を提供します。これは何らかの結果の `Observable` を返し、グリッド コンポーネントでそれを購読します。

**注**: Angular 5 より前では、`HttpClient` は `@angular/http` に配置され、`Http` という名前でした。

レコードの配列を含む JSON 応答を受け取るため、正しい形状のインターフェイスを定義して、observable で返されることを期待するデータの種類を指定しておくと便利です。型チェックは常に推奨され、後で発生する厄介な問題を回避するのに役立ちます。

```typescript
// northwind.service.ts

export interface NorthwindRecord {
    ProductID: number;
    ProductName: string;
    SupplierID: number;
    CategoryID: number;
    QuantityPerUnit: string;
    UnitPrice: number;
    UnitsInStock: number;
    UnitsOnOrder: number;
    ReorderLevel: number;
    Discontinued: boolean;
    CategoryName: string;
}
```

サービス自体は非常にシンプルで、1 つのメソッド `FetchData` で構成され、`Observable<NorthwindRecord[]>` を返します。何らかの理由（サーバーが利用不可、ネットワーク エラーなど）でリクエストが失敗した場合、`HttpClient` はエラーを返します。失敗した Observable をインターセプトし、エラーをエラー ハンドラーに渡す `CatchError` 演算子を活用します。エラー ハンドラーはエラーをログに記録し、安全な値を返します。

```typescript
// northwind.service.ts

@Injectable()
export class NorthwindService {
    private url = 'http://services.odata.org/V4/Northwind/Northwind.svc/Alphabetical_list_of_products';

    constructor(private http: HttpClient) {}

    public fetchData(): Observable<NorthwindRecord[]> {
        return this.http
            .get(this.url)
            .pipe(
                map(response => response['value']),
                catchError(
                    this.errorHandler('Error loading Northwind data', [])
                )
            );
    }

    private errorHandler<T>(message: string, result: T) {
        return (error: any): Observable<any> => {
            console.error(`${message}: ${error.message}`);
            return of(result as T);
        };
    }
}
```

アプリケーション モジュールで `HttpClientModule` とサービスの両方をインポートし、サービスをプロバイダーとして登録してください。

```typescript
// app.module.ts

import { HttpClientModule } from '@angular/common/http';
import { NorthwindService } from './northwind.service';

@NgModule({
    imports: [
        HttpClientModule
    ],
    providers: [
        NorthwindService
    ]
})
export class AppModule {}
```

サービスを実装した後、コンポーネントのコンストラクターに注入し、データを取得するために使用します。`ngOnInit` ライフサイクル フックは、初期リクエストをディスパッチするのに適した場所です。

**注**: 以下のコードで、サービスを購読する前に *records* プロパティを空の配列に設定している理由に疑問を持つかもしれません。Http リクエストは非同期であり、完了するまで *records* プロパティは *undefined* になります。その結果、グリッドがそれにバインドしようとするとエラーが発生します。既定値で初期化するか、`BehaviorSubject` を使用する必要があります。

```typescript
// my.component.ts

@Component({

})
export class MyComponent implements OnInit {

    public records: NorthwindRecord[];

    constructor(private northwindService: NorthwindService) {}

    ngOnInit() {
        this.records = [];
        this.northwindService.fetchData().subscribe((records) => this.records = records);
    }
}
```

```html

        <igx-column field="ProductId"></igx-column>
        <!-- rest of the column definitions -->
    </igx-grid>
```

## パフォーマンス（実験的）

<ApiLink type="GridToolbar" member="grid" label="Grid" /> の設計により、Angular が導入した Event Coalescing 機能を活用できます。この機能により、インタラクションと応答性の面でおおよそ **20%** 程度のパフォーマンス向上が可能になります。この機能は、`bootstrapModule` メソッドで `ngZoneEventCoalescing` および `ngZoneRunCoalescing` プロパティを **true** に設定するだけで、アプリケーション レベルで有効にできます。

```typescript
platformBrowserDynamic()
  .bootstrapModule(AppModule, { ngZoneEventCoalescing: true, ngZoneRunCoalescing: true })
  .catch(err => console.error(err));
```

<DocsAside type="info">

これは、`IgxGridComponent` に対してまだ実験的な機能です。つまり、グリッドで予期しない動作が発生する可能性があります。そのような動作が発生した場合は、当社の [Github]({GithubLink}/discussions) ページでお問い合わせください。

</DocsAside>

<DocsAside type="info">

これを有効にすると、`IgxGridComponent` に関連しない Angular アプリケーションの他の部分に影響する可能性があります。

</DocsAside>

</PlatformBlock>

## {Platform} グリッドのスタイル設定

> **注**:
> グリッドは **css grid layout** を使用しますが、これは **プレフィックスなしでは IE でサポートされていない**ため、正しくレンダリングされません。

<PlatformBlock for="WebComponents, Blazor, React">

定義済みテーマに加えて、利用可能な [CSS プロパティ](../grids/theming-grid.md)の一部を設定することで、グリッドをさらにカスタマイズできます。ヘッダーの背景とテキストの色を変更したい場合は、まずグリッドにクラスを設定する必要があります。

<PlatformBlock for="WebComponents">

```html
<igc-grid class="grid"></igc-grid>
```

</PlatformBlock>

<PlatformBlock for="React">

```tsx
<IgrGrid className="grid"></IgrGrid>
```

</PlatformBlock>

<PlatformBlock for="Blazor">

```razor
 <IgbGrid Class="grid"></IgbGrid>
```

</PlatformBlock>

次に、そのクラスに対して `--header-background` および `--header-text-color` CSS プロパティを設定します。

```css
.grid {
    --header-background: #494949;
    --header-text-color: #FFF;
}
```

</PlatformBlock>

<PlatformBlock for="Angular">

[**Angular**](https://angular.io/) では、[Autoprefixer](https://www.npmjs.com/package/autoprefixer) プラグインのおかげで、ほとんどのスタイルに暗黙的にプレフィックスが付与されます。

ただし、**グリッド レイアウト**にプレフィックスを付与するには、コメント `autoprefixer grid:on` を使用して [Autoprefixer](https://www.npmjs.com/package/autoprefixer) の **grid プロパティ**を有効にする必要があります。

作業を容易にするため、`src/styles.scss` ファイルにコメントを適用してください。

```scss
// src/styles.scss
@use "igniteui-angular/theming" as *;

// IMPORTANT: Prior to Ignite UI for Angular version 13 use:
// @import '~igniteui-angular/lib/core/styles/themes/index';
@include core();
/* autoprefixer grid:on */
@include theme($default-palette);
```

</PlatformBlock>

## 既知の制限

| 制限                                                                                                                                                                                            | 説明                                                                                                                                                                                                                                                                           |
| ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `percentage` および `px` で設定された列幅                                                                                                                                              | 現在、`%` と `px` を使用した列幅の混在はサポートしていません。                                                                                                                                                                                         |
| 型 `number` の列をフィルタリングしようとする場合                                                                                                                                                | `number` とは異なる値がフィルタリング入力に入力されると、不正なキャストにより `NaN` が返されます。                                                                                                                                                  |
| グリッドの `width` は列幅に依存しません                                                                                                                                             | すべての列の `width` は、グリッド自体のスパンを決定しません。これは親コンテナーの寸法、または定義されたグリッドの `width` によって決定されます。                                                                                                     |
| 親コンテナー内にネストされたグリッド                                                                                                                                                                       | グリッドの `width` が設定されておらず、定義済みの寸法を持つ親コンテナー内に配置されている場合、グリッドはこのコンテナーまで広がります。                                                                                                                                              |
| 列には許可される最小列幅があります。`--ig-size` CSS 変数に応じて、次のようになります: <br/>"small": 56px <br/> "medium": 64px <br/> "large ": 80px | 許可される最小値より小さい幅が設定されても、レンダリングされる要素には影響しません。対応する `--ig-size` に対して許可される最小幅でレンダリングされます。これにより水平仮想化で予期しない動作が発生する可能性があるため、サポートされていません。 |
| 行の高さは、現在ビュー内にレンダリングされていないセルの高さの影響を受けません。                                                                                                            | 仮想化のため、ビュー内にないカスタム テンプレートを持つ列（セルの高さを変更する列）は、行の高さに影響しません。行の高さは、関連する列がビュー内にスクロールされた場合にのみ影響を受けます。                                               |
